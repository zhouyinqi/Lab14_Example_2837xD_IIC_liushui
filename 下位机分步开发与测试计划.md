# 下位机分步开发与测试计划

## 1. 目标边界

本工程直接在已经验证 IIC/RTC 的 Lab14 工程中演进，目标是开发板级测试下位机固件。未来上位机通过一个“一键测试”命令读取测试清单、启动测试、接收逐项结果并生成报告。

本固件不是正式电机控制程序。上电默认不允许 PWM 和驱动使能；真实发波、300 V 母线、200 Arms 出流和 5 us 相序脉冲均不得进入默认一键测试。

## 2. HPD 大纲的权威判据

以 `HPD模组例行试验大纲-CS-260621-A.doc` 为准，已固化到 `include/Hpd_Test_Limits.h`：

| 项目 | 判据/参数 |
| --- | --- |
| 两套母线采样 | 60 V ±1 V |
| 两套线电压 | 200 V ±2% |
| 线电压频率 | 1500 Hz ±1% |
| 两套输出电流 | 200 A ±2% |
| 电流频率 | 200 Hz ±1% |
| 电流 THD | ≤3% |
| 三路 NTC 一致性 | 极差 ≤1 ℃ |
| NTC 对热电偶误差 | ≤2 ℃ |
| 驱动板固定占空比发波 | 1 s |
| 温度测试固定占空比发波 | 10 s |
| 相序测试脉冲 | 5 us |

原方案文本中的 `60 V ±2 V`、温度误差 `3 ℃`、相序脉冲 `3 us` 等数值与正式大纲不一致，后续实现不得继续使用。

## 3. 三类验证方式

| 类型 | 能证明什么 | 不能证明什么 |
| --- | --- | --- |
| 软件模拟 | 状态机、算法、阈值、协议、日志和上位机流程 | ADC 模拟前端、接线、传感器和功率回路 |
| 外部注入 | ADC 通道、模拟前端、标定、滤波和工程量换算 | 真实功率输出能力 |
| 试验台 | CAN/DI/DO、驱动反馈、真实电压电流、波形和保护链路 | 无 |

软件模拟结果必须标记为模拟结果，不能上报成真实硬件 PASS。

## 4. 分步实施

1. 测试固件骨架：统一测试 ID、结果、清单、命令邮箱、安全默认状态和 HPD 判据。接入现有 IIC/RTC。每次提交执行主机单元测试和 C2000 全量构建。
2. 板级配置：依据原理图建立 GPIO、ADC、PWM、CAN、SCI、SPI、EMIF 和故障信号映射。没有确认引脚的外设保持 `NOT_SUPPORTED`。
3. 低风险外设：实现系统时钟、Timer、GPIO、RAM、IIC、SCI、CAN 回环、SPI/EMIF 器件识别；形成安全一键测试序列。
4. 上位机协议：实现设备信息、测试清单、启动自动/单项测试、停止、实时结果和 CRC。先用串口或 CAN 选定一种链路跑通。
5. ADC 与软件模拟：实现母线、线电压、电流、NTC 的算法注入、滤波、RMS、频率、THD 和判据单元测试。
6. 外部注入：配置标定系数和工装输入点，验证 ADC 采样链路。结果明确标记 `EXTERNAL_INJECT`。
7. 低压 PWM：在 24 V、急停、驱动 Ready、故障输入、通信心跳和超时互锁全部通过后，测试两套驱动 1 s/10 s 固定占空比输出。
8. 高功率试验台：300 V、200 A、1500 Hz 和 5 us 相序测试使用独立授权命令；任何通信丢失或故障立即封锁 PWM。
9. 上位机一键测试：上位机按安全配置、外部注入配置、高功率配置三种方案运行，保存原始值、工程量、阈值、数据来源和结果。

## 5. 当前第一步的操作入口

当前尚未接入 CAN/SCI 命令解析。可在 CCS Expressions 中写入：

- `gBoardTestCommandMailbox.testId`：单项测试 ID，例如 `0x0303` 为 IIC/RTC。
- `gBoardTestCommandMailbox.command = 2`：启动单项测试。
- `gBoardTestCommandMailbox.command = 1`：启动当前安全自动序列。
- `gBoardTestCommandMailbox.command = 3`：停止测试并回到安全空闲。

观察 `gBoardTestManager.records` 获取逐项结果。当前 `SYS_STARTUP`、`GPIO` 和现有 `I2C_RTC` 已实现；其余项目明确返回 `NOT_SUPPORTED` 或 `SKIP`，不伪造 PASS。

### GPIO/程序灯测试

`BOARD_TEST_ID_GPIO`（`0x0200`）已实现。测试时 GPIO144/D34 与 GPIO143/D35 各保持 250 ms 交替输出，并检查两种输出组合和最终低电平的实际引脚回读。测试结束后两脚恢复为输入。

- `rawValue == 0x003F`：6 个回读点全部符合预期，结果为 `PASS`。
- 其他值：至少一个回读点异常，结果为 `FAIL`，错误码为 `0x0100`。
- 该结果验证 DSP GPIO 配置及引脚电平回读；没有光学传感器时，LED 是否实际发光仍需人工观察。

空闲主循环会持续调用程序灯心跳服务，GPIO144/D34 与 GPIO143/D35 每 250 ms 交替一次。该实现用于当前无上位机阶段的运行指示；接入通信协议和 CPU Timer 后将改为非阻塞定时驱动。

## 7. Flash 构建

Debug 与 Release 配置现在都使用 `CMD/2837x_FLASH_lnk_cpu1.cmd` 并定义 `_FLASH`。配置名称 `Debug` 仅表示保留调试符号，生成的 `.out` 同样是 Flash 地址映像，不再是 RAM 链接程序。

- Flash 启动入口：`0x080000`。
- `ramfuncs`：从 FLASHD 复制到 RAMLS0 后执行，包含 `InitFlash` 和延时函数。
- 默认输出：`Debug/Lab14_Example_2837xD_IIC_liushui.out`。
- 辅助十六进制输出：`Debug/Lab14_Example_2837xD_IIC_liushui.hex`。

使用 CCS 下载 `.out` 后可直接运行；若要求断电重启后自主运行，板卡启动模式还必须设置为 Flash Boot。

## 6. 下一步所需硬件资料

进入板级配置前需要主控板原理图或确认后的信号表，至少包含两套绕组的 ADC 通道、ePWM 对、驱动使能/Ready/Fault、DI3、DO3、CAN、SCI、SPI、EMIF 和外部器件型号。未经确认不得猜测引脚并下载到功率板。

实际 DSP 型号已确认为 `TMS320F28377D`，`.ccsproject` 和 `.cproject` 的 Debug/Release 目标器件均已相应修正。
